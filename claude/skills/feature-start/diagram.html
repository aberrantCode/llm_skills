homeradar-feature-start

Enforces a mandatory 5-step gate sequence before any HomeRadar feature implementation begins. Spec must be approved, plan must be written and approved — no self-approvals, no shortcuts.

Pre-Code Gate Sequence (all mandatory, strict order)
1
Find & Read the Spec Required first
Locate docs/features/<feature-name>.md. Read it fully before doing anything else — before creating worktrees, before reading code, before writing plans.
2
Verify Status: Approved HARD STOP if not approved Human sets this
Check the frontmatter: Status: Approved. If any other value — Draft, Review, In Progress — STOP immediately and surface it to the user. No exceptions. Proceeding on an unapproved spec wastes implementation effort.
3
Create Worktree Off latest dev
git worktree add .worktrees/<branch> -b feat/<branch>
  • Branch must be off latest dev
  • Use .worktrees/ directory (must be gitignored)
4
Write Implementation Plan B-style task contracts
Save to docs/plans/. Each task must include: CAP-ID, files to change, named test, done condition. Format from docs/workflow/SDLC.md Phase 2.
5
Get Plan Approved (Gate 2) Erik approves Then: ready to code
Present the plan for review. Do not write any implementation code until Gate 2 is cleared. Once approved → implementation can begin.

B-Style Task Contract Format (docs/plans/)
Each task in the implementation plan must specify:
CAP-ID
Capability identifier from the spec (e.g. HR-CAP-42)
Files
Specific files to create or modify
Named Test
Exact test name / file that verifies this task
Done Condition
Measurable condition — when is this task complete?
