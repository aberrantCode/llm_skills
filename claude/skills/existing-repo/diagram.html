Existing Repo

analyze · preserve conventions · setup guardrails · best practices

Core Principle
Understand before modifying. Existing repos have conventions, patterns, and history. Your job is to work within them — not reorganize them. Always run the full analysis before making any changes.

Structure Preservation Rules
Never Do These
  • Reorganize directory structure
  • Rename files "for consistency"
  • Add new patterns not in codebase
  • Change import styles
  • Change formatting without a formatter config
  • Add dependencies without checking equivalents
Always Do These
  • Read existing code first — understand patterns
  • Match existing naming conventions
  • Use existing utilities — don't reinvent
  • Follow existing test patterns
  • Preserve existing configs — only add, don't modify
  • Check if similar decision already exists
Gradual Guardrails Timeline (Don't Add All At Once)
  • Week 1
    Formatting — Prettier / Ruff
    Non-breaking, easy wins, builds trust
  • Week 2
    Linting — ESLint / Ruff
    Catches obvious issues without blocking workflow
  • Week 3
    Pre-commit hooks — Husky / pre-commit framework
    Automates weeks 1–2 checks on every commit
  • Week 4
    Commit message validation — commitlint
    Enforces conventional commits for team consistency
  • Week 5
    Type checking strictness — tsc / mypy
    Catches runtime errors at compile time
  • Week 6
    Security scanning — detect-secrets / CodeQL
    Prevents secret leaks and known vulnerabilities
