Detection
pm framework detection

Seven frameworks, priority-ordered

File-based markers only — no network calls, no authentication. First match wins. Records the detected framework in the cache as pm_framework:.

1. project-manager (docs/features/ + docs/plans/) 2. project-manager-partial (features only) 3. project-manager-seed (INITIAL_PROMPT.md) 4. backlog-md 5. tasks-md 6. plan-md 7. todo-md + github-issues (noted, not fetched) + changelog-driven (read-only cue) + none
area inference

Folder-derived task-ID prefixes

Top-level folders → 2–4-letter prefixes. Common-alias table recognises auth, ui, api, db, payments, etc. Confirmed via AskUserQuestion on first run; cached for subsequent runs.

src/auth → AUTH src/ui → UI src/api → API packages/pay → PAY services/core → CORE reporting → REPO (truncated) user-profile → UP (acronym) <ambiguous> → GEN (fallback) <reserved> → SEC · ARCH · TEST · DOC
infra-gap scan iter-2

Observations for task sequencing

Lightweight existence checks during Step 2: no test runner, no CI, no linter config. Recorded in the cache as observations:. Step 7 surfaces one as an FYI hint above the top-three prompt when relevant.

no test runner configured no CI — manual validation only no lint/format config [one line, not a lint pass]
Prioritisation
weighted heuristic

Eight scoring signals, additive

Pending tasks get a score; top three go to the user. Weights are overridable via a priority_weights: block in the cache for per-project tuning.

+40 blocks one or more tasks +35 security / SEC- prefix +30 build-breaking +25 failing-tests target +20 already in-progress +15 priority: p1 +5 priority: p2 + age bonus (log-scaled) -3 stub (desc < 20 chars) -5 priority: p3
implicit-blocked penalty iter-2

-15 computed from {blocks} tags

If task B is listed in task A's {blocks: ..., B} tag, the skill infers B is blocked by A. No need for an explicit {blocked} tag. The penalty evaporates when A flips to - [x].

AUTH-001: {blocks: AUTH-002} → AUTH-002 gets -15 (implicitly blocked) Keeps blocker + blocked task from competing for top-three.
stale penalty iter-2

-25 when referenced file is gone

If a task's description mentions a file path that doesn't exist, the skill flags it stale and drops it out of the top-three. Stale tasks are never auto-closed — only the user decides whether to retire or rewrite.

LEGACY-001: Refactor src/old/thing.ts → src/old/thing.ts missing → marked stale, -25 penalty → excluded from top-three → checkbox stays unchecked
Cache + Freshness
fingerprint-based cache

SHA-256 of key files, not mtime

Stores hashes of package.json, docs/features/*, backlog.md, etc. Recomputed on every run — if any differ, the cache is stale and the user is offered a refresh.

backlog.md: 3a7e1f... package.json: c29b84... docs/features/*: d8a1f0... README.md: 81ee44... Lockfiles skipped (churn too much). mtime never trusted.
update flow

/what-next update · forced refresh

Dedicated command (and alias /what-next-update) for forced re-analysis + backlog reconciliation. Any task whose referenced file is missing is marked stale; a one-line diff summary is printed before the top-three prompt.

Areas changed: + MOBILE Tasks added: 4 Archived: 2 Marked stale: 1 (LEGACY-001) Fingerprints refreshed: 2/2
current_task resumability

Recover mid-task across sessions

When Step 8 hands off, the chosen task ID is written to current_task: in the cache. A later run sees the work is in progress and can resume cleanly rather than presenting a fresh top-three.

current_task: AUTH-004 → Step 7 offers: - "Resume AUTH-004" - "Show me top three instead" → Completion clears the field.
Task-store lifecycle
backlog.md format

Checklist with area-scoped IDs

Pure markdown — no schema lock-in. Tags are inline with curly braces. Completed items move to backlog-archive.md under date headings; IDs are never reused.

- [ ] AUTH-001: Reset flow {priority: p1} - [x] UI-008: Button alignment {priority: p2} - [ ] SEC-001: JWT audit {security} - [ ] LEGACY-001: Refactor src/old/thing.ts {stale: target-missing}
bootstrap (bare repo)

Four options via AskUserQuestion

When a repo has no task store, the user picks the path forward. No silent file creation.

1. Run /project-manager setup 2. Create a backlog.md [Recommended] 3. Run a repo-wide code analysis 4. I'll tell you the tasks myself
code-analysis bootstrap

Four specialist agents in parallel

Option 3 of the bootstrap. Spawns read-only audits; findings become backlog items with reserved prefixes.

code-reviewer → {AREA}-NNN security-reviewer → SEC-NNN architect → ARCH-NNN tdd-guide → TEST-NNN Each returns ≤ 25 findings as - severity: file:line desc.
Delegation + Hand-off
project-manager delegation iter-2

Preview + hand-off, not punt

When project-manager state exists, the skill still reads plans, applies the heuristic to todo rows, and surfaces the top 3 concrete plan tasks alongside /review-tasks and /continue-tasks options. The user picks; backlog.md is never created.

AskUserQuestion: (a) auth P1-T3 — rate-limiting (b) auth P2-T1 — reset token schema (c) payments P1-T1 — Stripe SDK (d) /review-tasks (full snapshot) (e) /continue-tasks (PM loop)
specialist agent hand-off

Keyword → agent mapping

Chosen task is classified by keywords in its description, then routed to the right specialist. Mirrors project-manager's role table for consistency.

feature, add, implement → tdd-guide review, quality → code-reviewer security, SEC- → security-reviewer architecture, design → architect build, types, tsc → build-error-resolver e2e, playwright → e2e-runner docs, readme → doc-updater cleanup, dead code → refactor-cleaner plan, spec → planner anything else → general-purpose
in-place status mutation

Mark in-progress where it belongs

For backlog.md tasks: (in-progress) is appended to the line. For project-manager plans: the row's Status: column is updated in place. current_task is set in the cache either way.

backlog-md mode: - [ ] AUTH-001: ... (in-progress) project-manager mode: | P1-T3: ratelimit | ... | in-progress |
