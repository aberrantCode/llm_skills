Proposed — iteration 3+

Broader eval coverage Proposed

Current harness has 4 evals. Add coverage for the branches not yet exercised.

  • Eval 5: tasks.md framework (distinct from backlog.md)
  • Eval 6: monorepo with apps/* + packages/* — exercises collision-handling in area inference
  • Eval 7: large backlog (50+ items) — stress-tests the priority heuristic and top-3 selection stability
  • Eval 8: cache-hit path where fingerprints match — verifies the fast path doesn't re-analyse
  • Eval 9: current_task resumability — skill started eval 1, session ends, new session resumes cleanly
Proposed — iteration 3+

Description optimization loop Proposed

Use skill-creator/scripts/run_loop.py to auto-optimise the SKILL.md description: field against 20 trigger eval queries. Currently the description was hand-written; it likely under-triggers on less obvious phrasings like "I lost the thread on this project" or "dust off this repo".

  • Generate 20 trigger queries (8–10 should-trigger, 8–10 near-miss should-not-trigger)
  • Run up to 5 optimisation iterations with extended-thinking proposals
  • Apply the best description (chosen by held-out test score, not train score)
Proposed — iteration 3+

Real-repo integration tests Proposed

Synthetic fixtures cover the known decision branches. Real-repo tests would catch the ones we didn't think to write. Pick 3–5 real projects with diverse PM styles and run /what-next against them in dry-run mode.

  • Pure-backlog.md repo (e.g., small OSS tool)
  • project-manager-heavy repo (dogfooded on this very archive)
  • GitHub-Issues-driven repo (exercises the "detected but not fetched" path)
  • Large monorepo with Lerna/Turbo
  • Non-code repo (research notes, content) to verify graceful degradation
Proposed — later

External tracker bridges Proposed

Currently github-issues, Jira, Linear are noted in the cache but not fetched. If authentication patterns stabilise, optionally surface open issues/tickets in the top-three alongside backlog items. Must remain opt-in; file-based detection stays the default.

  • gh issue list — uses gh CLI if installed + repo remote present
  • Linear/Jira — via their MCP servers when available
  • Explicit user consent required per project; write chosen source to docs/what-next.md
