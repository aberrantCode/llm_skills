Skill · What Next?

Solution Architecture

Three tiers of state: the archive is the source of truth; the global profile deploys the skill to every project; the target repo is where the skill actually runs. The skill itself is progressive-disclosure markdown — references load only when their branch fires.

Skill Bundle — what ships inside the skill directory
Level 1 · always in context

SKILL.md

claude/skills/what-next/SKILL.md

Frontmatter + master decision flow. Under 500 lines. Body loads whenever the skill triggers; references load only when their branch fires.

Level 2 · lazy

references/

references/*.md

Five specialised docs: fingerprint algorithm, area inference, priority weights, backlog templates, what-next.md schema. Loaded only when the corresponding decision branch fires.

Level 2 · trigger

commands/

commands/what-next.md
commands/what-next-update.md

Thin slash-command wrappers that invoke the skill. Pushed to ~/.claude/commands/ for global availability.

Level 3 · development-time only

evals/

evals/{fixtures,harness,benchmarks}/

Committed fixtures + reusable grader + benchmark history. Never read at skill-runtime; used only when a developer runs a new evaluation iteration.

Documentation

diagram.html + diagrams/

diagram.html (decision flow)
diagrams/{solution,features,plan}.html

Decision-flow diagram lives at the skill root per repo convention; supporting visuals live in diagrams/. All self-contained HTML, openable in a browser.

Registry

manifest.json (repo-level)

manifest.json

Remote installer reads this to know which skills exist and what category each belongs to. what-next is registered under Foundations & Workflow.

Runtime state in the target repo
Cache

docs/what-next.md

docs/what-next.md (created on first run)

Machine-readable YAML frontmatter (pm_framework, areas, fingerprints, observations, current_task) plus a human-readable project snapshot. Fingerprint check at Step 1 decides whether to re-analyse.

Task store

backlog.md + backlog-archive.md

repo root

Used only when no project-manager convention exists. Checklist format with area-scoped IDs; completed items move to backlog-archive.md under a ## YYYY-MM-DD heading. IDs never reused.

Delegated

docs/features/ + docs/plans/

project-manager's own state

When detected, /what-next reads these, applies the priority heuristic to todo rows, surfaces top 3, and lets the user pick a task (direct hand-off) or /review-tasks / /continue-tasks (full PM loop). Never creates a parallel backlog.md.

Cross-cutting invariants
Invariant · user interaction

AskUserQuestion only

Every branching decision — which bootstrap option to take, which top-three task to pick, whether to refresh a stale cache — goes through the AskUserQuestion tool. No prose prompts. Option labels and descriptions carry the ranking reasons.

Invariant · no competing stores

Delegate, don't duplicate

If project-manager state exists, /what-next reads and surfaces it, but never creates a parallel backlog.md. Plans remain the single source of truth; mutation happens in place.

Invariant · no destructive actions

Stale, not auto-closed

Tasks referencing deleted files get a -25 priority penalty and a stale marker. The skill never closes them on the user's behalf — only the user decides whether to retire or rewrite.
