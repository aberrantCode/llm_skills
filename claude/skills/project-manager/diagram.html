Skill Reference · project-manager

Project Manager

Self-driving project pipeline. Feature specs → Plans → Tasks → Agents → Completion. Markdown files in docs/ are the single source of truth, persisting across sessions.

Available Commands
/continue-tasks
Full Orchestration Loop
Bootstrap → Plan → Find Next Task → Write Task File → Spawn Agent → Monitor Completion. Main driver.
/review-tasks
Dry-Run Status Report
Read-only project snapshot. Counts specs, plans, task statuses, active files, issues. No file modifications.
/update-tasks
Sync Completed Tasks
Scan active task files for Completion sentinels. Archive completed tasks, update plan statuses. Idempotent.
/reinit
Archive Legacy + Launch
Archive old plans/tasks, normalize feature specs to template, then run /continue-tasks. Recovers from inconsistent state.

Directory Conventions
docs/
  INITIAL_PROMPT.md  ← source of truth (never modified)
  features/  ← feature specs (final authority on scope)
    {feature-slug}.md
  plans/  ← one plan per feature spec
    {feature-slug}-plan.md
  tasks/
    active/  ← at most one file per work stream
      {feature-slug}-p{N}-t{M}.md
    archive/  ← completed tasks move here immediately
  issues/  ← logged failures & blockers
  guides/  ← supporting docs, architecture notes

Error Recovery Loop — Up to 5 Iterations
01
Read failure message from sentinel Error: field
02
Diagnose root cause
03
Add 1–3 corrective tasks before failed task
04
Increment failures counter on plan
05
If ≥5: Pause, write issue file, ask user
Working Principles
  • One task at a timeParallel tasks in the same feature create merge conflicts and make failure harder to diagnose.
  • Plans are living checklistsA plan file is both the specification and the progress tracker.
  • Task files carry full contextAn agent reads only its task file. Embed all relevant spec excerpts.
  • Archive aggressivelyCompleted tasks move immediately to archive/. Active/ has at most one file per stream.
  • Specs before plans, plans before tasksNever generate a task without an approved spec. Pipeline flows one direction only.
  • Spec authority ruleSpecs are the final authority. Never silently update specs during implementation.
