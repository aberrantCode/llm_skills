Retro-fit Spec
[XX-CAP-NN]

sdlc.md registry → capability numbering → isolated commit

When a HomeRadar feature spec's Capabilities section has bullets without [XX-CAP-NN] prefixes, this skill retro-fits them before any implementation work begins. It looks up the 2-letter feature abbreviation, numbers each capability sequentially, updates the spec's status, and commits the retro-fit in isolation — never bundled with implementation changes.
inputs & outputs
Inputs
  • Feature spec file with un-prefixed capability bullets
  • docs/workflow/SDLC.md — Feature Abbreviation Registry table
  • Feature topic or known 2-letter abbreviation (e.g. PT)
Outputs
  • Spec with every capability prefixed: [XX-CAP-01], [XX-CAP-02]…
  • Spec Status field set to In Progress
  • Isolated commit: docs: retro-fit CAP-IDs to <spec-name>.md

commit formats
# After retro-fit — ISOLATED commit, never bundled with implementation
docs: retro-fit CAP-IDs to <spec-name>.md

Refs: XX-CAP-01
Spec: docs/features/<spec-name>.md

# Until retro-fitted — interim format using filename#capabilities
Refs: <spec-filename>#capabilities
Spec: docs/features/<spec-name>.md
hard rules
Never bundle
Retro-fit commits must be isolated — never combined with implementation commits. One retro-fit, one commit.
Sequential from 01
Number capabilities starting at 01 in document order: [PT-CAP-01], [PT-CAP-02]…
Interim format
While un-retro-fitted, use <filename>#capabilities in Refs — never a bare CAP-ID that doesn't exist yet.
Registry first
Always look up the 2-letter abbreviation in SDLC.md before numbering — don't guess or invent abbreviations.
