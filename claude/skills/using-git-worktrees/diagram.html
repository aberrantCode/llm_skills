Using Git
Worktrees

superpowers:using-git-worktrees

Creates an isolated workspace sharing the same repository — smart directory selection, mandatory gitignore safety verification, auto-detected project setup, and clean baseline confirmation before any work begins.

Directory Selection Priority
Priority order — first match wins
Priority 1 ── .worktrees/ exists HIGHEST use it (verify gitignored)
2 ── worktrees/ exists HIGH use it (verify gitignored)
3 ── CLAUDE.md preference MED use without asking
4 ── Ask user FALLBACK project-local or ~/.config/superpowers/worktrees/

Tiebreak ─── Both exist? .worktrees/ always beats worktrees/
Safety Verification
Project-Local Path MUST verify

Run git check-ignore -q .worktrees before creating the worktree.

  • Passes → proceed immediately
  • Fails → add line to .gitignore, commit the change, then proceed

Why critical: Without this, worktree contents get tracked — pollutes git status, risks committing thousands of generated files.

Global Path No check needed

Path: ~/.config/superpowers/worktrees/<project>/<branch>

  • Entirely outside project directory
  • Git never tracks it
  • Skip all gitignore verification
  • Proceed directly to worktree creation
Example Session
using-git-worktrees — feature/auth
# Step 1: Check existing directories
$ ls -d .worktrees 2>/dev/null
.worktrees

# Step 2: Verify ignored
$ git check-ignore -q .worktrees && echo "ignored" || echo "NOT ignored"
ignored ✓ — safe to proceed

# Step 3: Create worktree
$ git worktree add .worktrees/auth -b feature/auth
Preparing worktree (new branch 'feature/auth')
HEAD is now at a3f19e2 latest commit

# Step 4: Auto-detect setup
$ cd .worktrees/auth && npm install
added 847 packages in 12s

# Step 5: Baseline verification
$ npm test
✓ 47 passing (3.2s)

Worktree ready at /project/.worktrees/auth
Tests passing (47 tests, 0 failures)
Ready to implement feature/auth

Key Rules
🚫
Never skip gitignore check
For project-local paths, always verify with git check-ignore before creation. Fix immediately if missing.
🚫
Never assume directory
Follow priority strictly: existing dir → CLAUDE.md → ask user. Never decide arbitrarily.
🚫
Never hardcode setup commands
Auto-detect from project files. Different projects use different tools — hardcoding breaks cross-project use.
✓
Always verify baseline
Run tests after setup. Can't distinguish new bugs from pre-existing issues without a clean baseline.
✓
Fix immediately
Directory not gitignored? Add to .gitignore and commit right away — don't defer or warn and continue.
✓
Report failures, ask to proceed
If baseline tests fail, report the failures and explicitly ask whether to continue before doing anything else.
