Doc Co-Authoring

context gathering → section-by-section refinement → reader testing

Inputs
  • Doc topic, type, audience, desired impact
  • Any existing template or shared doc
  • User's context dump — org, history, constraints
  • Links to Slack/Teams/Drive (if integrations available)
→
Output
  • Polished document that works for its readers
  • Passes Reader Claude testing (no blind spots)
  • Every section refined through iteration
  • Verified: works when pasted into Claude
1
Context Gathering
until enough is known
meta-questions (upfront)
What type of document is this?
Who is the primary audience?
What's the desired impact?
Any template or format to follow?
Other constraints or context?
info dump (encourage freely)
Background on project / problem
Related team discussions or threads
Why alternatives aren't being used
Org context: team dynamics, past incidents
Timeline pressures, technical dependencies
After dump, ask 5–10 numbered clarifying questions. User can answer in shorthand — "1: yes, 2: see #channel, 3: no because backwards compat". If integrations (Slack/Drive) are available, use them to pull context. Exit when questions show understanding — when edge cases and trade-offs can be asked about without needing basics explained.
2
Refinement & Structure
one section at a time — repeat until all done
Start with the section that has the most unknowns (for decision docs: the core proposal; for specs: the technical approach). Summaries are written last. Use str_replace for all edits — never reprint the full document.
a
Clarifying questions for this section
b
Brainstorm 5–20 options to include
c
User curates: keep / remove / combine
d
Gap check: anything important missing?
e
Draft section via str_replace placeholder
f
Iterative refinement until satisfied
After 3 consecutive iterations with no substantial changes, ask if anything can be removed without losing information. At 80%+ completion, re-read the entire document for flow, redundancy, contradictions, and "slop" (generic filler sentences that carry no weight).
3
Reader Testing
fresh Claude with no context
01
Predict
Generate 5–10 realistic questions a reader would ask when trying to discover or understand this doc.
02
Test
Invoke sub-agent with just the document + each question. No context from this conversation.
03
Check
Ask Reader Claude for ambiguity, false assumptions, contradictions in the doc.
04
Iterate
Fix any gaps found. Loop back to Stage 2 for problematic sections. Repeat until Reader Claude answers consistently correctly.
If sub-agents are not available, provide the user with step-by-step manual testing instructions using a fresh Claude.ai tab.
decision points
Workflow accepted?
Yes → proceed to Stage 1
No → work freeform
Enough context?
Yes → move to Stage 2
No → ask more questions
Section done?
Yes → next section
No → more refinement (f→f→f)
Reader test passes?
Yes → final review
No → fix gaps, re-test
effective guidance
Context Management
Don't let context gaps accumulate — address them as they come up. Proactively ask when something mentioned is unfamiliar.
Artifact Management
Use create_file for full drafts. Use str_replace for all edits. Never use artifacts for brainstorm lists — that's conversation.
Quality over Speed
Don't rush through stages. Each iteration should make meaningful improvements. The goal is a document that actually works for readers.
