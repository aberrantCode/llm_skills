Data Modeling Patterns
Denormalized Document (embed for reads)
// posts/{postId}
{
  id: "post-1",
  title: "Hello World",
  authorId: "user-123",
  // Denormalized — avoids join
  authorName: "Alice",
  authorAvatar: "https://...",
  likeCount: 42,  // aggregated
  tags: ["js", "react"],
  createdAt: Timestamp
}
Subcollection (unbounded lists)
// posts/{postId}/comments/{commentId}
{
  id: "comment-1",
  text: "Great post!",
  authorId: "user-456",
  authorName: "Bob",
  createdAt: Timestamp
}

// Use subcollections for:
// • Unbounded lists (comments, messages)
// • Different access patterns
// • Data that grows independently
Real-time Listeners & Transactions
onSnapshot — React Hook
const unsubscribe = onSnapshot(
  doc(db, 'posts', postId),
  (snapshot) => {
    if (!snapshot.exists()) {
      onData(null); return;
    }
    onData({
      id: snapshot.id,
      ...snapshot.data()
    });
  },
  onError
);
// Cleanup in useEffect return:
return unsubscribe;
Transaction — like/unlike
await runTransaction(db,
  async (tx) => {
    const postSnap =
      await tx.get(postRef);
    const likeSnap =
      await tx.get(likeRef);

    if (likeSnap.exists())
      throw new Error('Already liked');

    tx.set(likeRef, { createdAt });
    tx.update(postRef, {
      likeCount: increment(1)
    });
  });
Anti-Patterns
✗
No security rules in production — test mode allows anyone to read/write everything; always deploy rules
✗
Deep nesting — keep documents flat; max 2-3 levels; deeply nested data is hard to query and secure
✗
Unbounded arrays — arrays can't be paginated; use subcollections for lists that grow
✗
Large documents — max 1MB per document; split large objects across subcollections or Storage
✗
No offline handling — enable IndexedDB persistence for mobile/PWA; check metadata.fromCache
✗
No emulator testing — always test rules against the Firestore emulator before deploying to production
