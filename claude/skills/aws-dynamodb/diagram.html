CRUD + GSI Query Patterns
Put with condition
await docClient.send(new PutCommand({
  TableName: TABLE_NAME,
  Item: {
    PK: `USER#${userId}`,
    SK: 'PROFILE',
    EntityType: 'User',
    email, name,
    createdAt: now,
  },
  // Prevent overwrite
  ConditionExpression:
    'attribute_not_exists(PK)'
}));
GSI query by order ID
const result = await docClient.send(
  new QueryCommand({
    TableName: TABLE_NAME,
    IndexName: 'GSI1',
    KeyConditionExpression:
      'GSI1PK = :pk',
    ExpressionAttributeValues: {
      ':pk': `ORDER#${orderId}`
    }
  })
);
// Returns order + all items
TransactWrite — Atomic Multi-Item Operation
await docClient.send(new TransactWriteCommand({
  TransactItems: [
    // Create order
    { Put: { TableName, Item: orderItem,
        ConditionExpression: 'attribute_not_exists(PK)' } },
    // Atomic counter increment
    { Update: { TableName, Key: keys.user(userId),
        UpdateExpression:
          'SET orderCount = if_not_exists(orderCount, :zero) + :inc',
        ExpressionAttributeValues: { ':zero': 0, ':inc': 1 } } },
    // Add each order item
    ...items.map((item, i) => ({
      Put: { TableName, Item: {
        PK: `ORDER#${orderId}`, SK: `ITEM#${i}`,
        EntityType: 'OrderItem', ...item
      }}
    }))
  ]
}));
Anti-Patterns
✗
Scan operations — always use Query with partition key conditions; Scan reads entire table
✗
Hot partitions — distribute writes with high-cardinality PK values; monotonically increasing keys cause hot spots
✗
Too many GSIs — each GSI duplicates data and costs write capacity; design access patterns first
✗
No condition expressions — always validate with ConditionExpression to prevent overwrites and race conditions
✗
Multi-table design without reason — single-table is preferred; only split when access patterns truly diverge
