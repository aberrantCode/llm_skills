Create Feature Spec

comprehensive feature specification from a single sentence · iterative refinement · /create-feature-spec

Input & Purpose
Input: A single sentence describing a new feature.
Output: A comprehensive, multi-section feature specification saved to /docs/features/{FEATURE_NAME}.md
Trigger: /create-feature-spec
Iterative
AskUserQuestion at each major decision point. Multiple refinement rounds until user approves.
Comprehensive
Don't skip sections. If optional, explain why it's not applicable. All REQUIRED sections get substantive content.
Grounded
Reference existing code, patterns, and components from the codebase. Not abstract — based on reality.
process
The 7-Step Process
1
Initial Analysis
Read FEATURE_SPECIFICATION.md template. Parse the sentence to identify core problem + proposed solution. Identify which sections are applicable.
2
Gather Context (tools, not questions)
Search codebase with Glob/Grep. Find similar features, existing components, services, data models, API endpoints, architecture patterns from CLAUDE.md.
3
Ask Clarifying Questions
AskUserQuestion: 3–4 strategic questions. Scope boundaries · user personas · technical approach · integration points. First action in any session.
4
Create Initial Draft
Generate complete spec following template. Fill ALL required sections with substantive content. Use codebase examples to ground technical details.
Continuing the Process
5
Iterative Refinement
After initial draft: AskUserQuestion about sections needing more detail, missing aspects, structural changes. Each iteration improves specific sections — not full rewrites.
6
Quality Checklist
Before marking complete: all required sections filled · user stories clear · architecture shown · data model complete · API endpoints detailed · success metrics measurable.
7
Save Output
Save to /docs/features/{FEATURE_NAME}.md. Use kebab-case filename matching the feature name in the document header.

Quality Checklist (Before Complete)
  • ✓Required sections have substantive content
  • ✓User personas & stories are clear and actionable
  • ✓Architecture section shows fit into existing system
  • ✓Data model includes schema and validation rules
  • ✓API spec has detailed request/response examples
  • ✓Implementation phases with deliverables are clear
  • ✓Success metrics are measurable and specific
  • ✓Open questions & future enhancements captured
  • ✓Related docs & ADRs are linked
