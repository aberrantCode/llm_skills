Workspace Analysis

workspace skill — multi-repo topology, API contracts, cross-repo context

The Problem
Claude Code operates in isolation across repos — missing API contracts, shared types, full architecture, and cross-repo dependencies. This causes duplicate type definitions, API mismatches, breaking changes caught only at runtime. Solution: dynamic workspace analysis with auto-refreshed artifacts.

Analysis Protocol — 4 Phases
1
Topology Discovery
Identify workspace root, classify type, list all modules/apps/packages, detect tech stack per module, identify entry points.
~30 seconds
2
Dependency Graph
Map internal deps (workspace imports), API relationships (fetch/axios calls), database connections (Prisma/Drizzle/SQLAlchemy).
~60 seconds
3
Contract Extraction
Parse OpenAPI/GraphQL/tRPC/Protobuf/TypeScript/Pydantic/Zod. Priority: generated specs → schema defs → type exports → inferred.
~45 seconds
4
Key File Identification
Routes (HIGH), Types (HIGH), API clients (HIGH), Config (MEDIUM), Entry points (MEDIUM), Tests (LOW/on-demand).
~30 seconds

Freshness indicators: 🟢 Fresh (synced <24h, no source changes) · 🟡 Stale (sources changed since sync) · 🔴 Outdated (>7 days) · ⚠️ Drift (validation found inconsistencies).
Commands: /analyze-workspace (full ~2min, first setup) · /sync-contracts (incremental ~15s, run frequently) · /workspace-status (quick check). Multi-repo access: use sibling directory convention (../myapp-backend/) or set up workspace symlinks in .workspace/repos/.
