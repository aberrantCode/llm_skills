Explain
Code

analogy · diagram · walkthrough · gotcha — always all four

Use when explaining how code works, teaching about a codebase, or when a user asks "how does this work?". Every explanation follows the same four-element structure — always, in order, without skipping. Keep explanations conversational. For complex concepts, use multiple analogies.
input → output
Input
  • Any code snippet the user is confused about
  • A function, class, module, or system
  • Question like "how does X work?"
  • Codebase the user is trying to understand
→
Output
  • Everyday-life analogy that makes it click
  • Visual diagram of flow, structure, or relationships
  • Step-by-step code walkthrough
  • One "gotcha" — the most common misconception
the four elements — always in this order
1
Analogy
compare to something from everyday life
Pick a concrete, familiar thing that maps onto the code's behaviour. The analogy should explain the core mechanic, not just the name. For complex concepts, stack multiple analogies from different angles.
A closure is like a backpack — the function carries its own environment with it wherever it goes, even after it's left the room where it was packed.
2
Diagram
show the flow, structure, or relationships
Draw a visual representation — a flowchart, box diagram, sequence, or ASCII art sketch. Show how the pieces connect, not just what they're named. Favour spatial clarity over completeness; a simple diagram that illuminates beats a complete one that overwhelms.
An event loop diagram showing the call stack, microtask queue, and macrotask queue as three lanes, with arrows showing which drains first.
3
Walkthrough
step-by-step through what happens
Trace the code execution in order, explaining each step in plain language. Reference the actual code — use line N or function name anchors. Keep it conversational: "first this happens, then that, because…"
"Line 3: useEffect fires after render. Line 7: we return a cleanup — React calls this before the next effect runs or the component unmounts."
4
Gotcha
the common mistake or misconception
Surface the one thing most people get wrong about this code. Frame it as "people expect X, but actually Y". This is the most memorable part of the explanation — make it specific and precise.
"People expect async/await to make code run in parallel — it doesn't. It just pauses this function while other already-running tasks continue. You need Promise.all() for actual concurrency."
style guidance
Conversational tone. Write as if explaining to a colleague over coffee, not writing documentation. Contractions, parenthetical asides, and "notice that…" are fine.
Multiple analogies for complexity. One analogy rarely covers every angle of a complex concept. Offer 2–3 from different perspectives — mechanical, spatial, social — and let the reader pick the one that sticks.
