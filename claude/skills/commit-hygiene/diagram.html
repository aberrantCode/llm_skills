Commit Hygiene

atomic commits · pr size limits · commit thresholds · stacked prs

Core Philosophy
Atomic Commits
One logical change per commit. Each commit should be self-contained and deployable. If you need "and" to describe it — split it.
Small PRs Win
<400 lines = reviewed in <1 hour. >1000 lines = likely rubber-stamped or abandoned. Smaller PRs = faster reviews, fewer bugs, easier reverts.
Commit Early & Often
Working code? Commit it. Test passing? Commit it. Don't wait for "done" — commit at every stable point.

  • ✓ Feature / function complete
  • ✓ Refactoring done

  • ✓ About to switch context
  • ✓ >5 files changed
  • ✓ >200 lines changed
  • ✓ About to try something risky
Splitting Strategies
Strategy 1: By Layer
API + DB + Frontend + tests (1 commit)
1: migration → 2: model → 3: endpoint → 4: component → 5: tests
Strategy 2: By Feature Slice
All CRUD for users (1 commit)
1: create → 2: read → 3: update → 4: delete
Strategy 3: Refactor First
feature + refactoring mixed
1: refactor helpers → 2: add feature using helpers
PR Size vs Defect Rate (Research)
15%
PRs <200 lines
23%
PRs 200–400 lines
40%+
PRs >400 lines
Review quality drops sharply after 200–400 lines. Large PRs get "LGTM" rubber stamps, not real reviews.
Stacked PRs for Large Features
PR #1: DB schema (200 lines) → PR #2: API (250 lines) → PR #3: Frontend (300 lines). Each is reviewable independently. Merge in order.
Red Flags — Stop and Split
Commit message needs "and" >10 files in one commit >400 lines in one commit Mix of features + fixes + refactors "WIP" or "Updates" commit messages "I'll clean this up later"
