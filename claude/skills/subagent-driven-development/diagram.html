Skill Reference

Subagent-Driven
Development

superpowers:subagent-driven-development

Execute implementation plans by dispatching a fresh subagent per task — eliminating context pollution — then enforcing a mandatory two-stage review (spec compliance first, code quality second) before marking any task complete.

Inputs & Outputs
Inputs
Implementation plan file (e.g. docs/plans/*.md)
Task list with full text & context for each task
Optional: scene-setting context (project purpose, conventions)
Outputs
Completed, committed code for all tasks — tests passing
All tasks marked complete in TodoWrite
Final code review sign-off on entire implementation
Branch ready for finishing-a-development-branch

Phase-by-Phase Detail
phase 0
Setup
Read plan once, extract everything upfront Controller
Read the plan file once. Extract all tasks with their full text, context, and dependencies. Create a TodoWrite list with every task. The controller curates exactly what each subagent needs — subagents never read the plan file directly.
phase 1
Implement
Dispatch Implementer Subagent Fresh context
Provide full task text + scene-setting context. The implementer gets complete information upfront — no file-reading overhead. If it asks questions, answer completely before letting it start work. The implementer writes code, tests (TDD), commits, and self-reviews.
Implementer asks questions?
yes → answer fully, then re-dispatch no → proceed to implementation
phase 2
Spec Gate
Gate 1 — Spec Compliance MUST pass before code quality review
Dispatch a Spec Compliance Reviewer subagent with the git SHAs. This reviewer checks two things: missing (required behavior not implemented) and extra (unrequested additions that weren't specced). Both are failures.
↺ Issues found → same Implementer subagent fixes → re-review until ✓ ✓ All clear → advance to Gate 2
phase 3
Quality Gate
Gate 2 — Code Quality Only runs after Gate 1 is ✓
Dispatch a Code Quality Reviewer subagent (see code-quality-reviewer-prompt.md). Reviews architecture, naming, test quality, edge cases, and code smells — entirely separate concern from spec compliance. Never skip this gate even if "the task is simple."
↺ Issues found → same Implementer subagent fixes → re-review until ✓ ✓ Approved → mark task complete in TodoWrite
phase 4
Loop
Advance to next task Sequential only
One task at a time — never dispatch multiple implementation subagents in parallel (file conflicts). Each new task gets a fresh subagent with its own extracted context. Continue until all tasks in TodoWrite are marked complete.
phase 5
Finish
Final code review + finishing-a-development-branch Completion
Dispatch a final code reviewer across the entire implementation (not per-task). Then invoke superpowers:finishing-a-development-branch to verify tests, present options (merge/PR/keep/discard), and clean up the worktree.
Mandatory Review Order
Starting code quality review before spec compliance is confirmed is a hard violation.

Key Rules
🚫
Never skip reviews
Both spec compliance AND code quality are mandatory per task — every single time, no exceptions for "simple" tasks.
🚫
Never parallel-implement
Multiple implementation subagents in parallel cause file conflicts. Tasks execute one at a time in strict sequence.
🚫
Never skip re-reviews
Reviewer found issues → fix → review again. "Close enough" is not done. Don't advance until the reviewer explicitly approves.
✓
Always extract upfront
Controller reads the plan once and extracts all task text. Subagents receive their task as provided text — never sent to read plan files themselves.
✓
Always answer questions
If the implementer asks questions before starting, answer clearly and completely. Don't rush them into implementation before they have what they need.
⚡
Fresh context per task
Each implementer subagent starts with zero context from prior tasks. This eliminates context pollution — bugs from Task 3 can't bleed into Task 4.
