AWS Aurora

Aurora Serverless v2 auto-scaling, RDS Proxy for connection pooling, Data API for HTTP access — production-grade PostgreSQL on AWS.

Serverless v2 RDS Proxy Data API
Connection Strategies
Strategy 1 — Recommended
RDS Proxy
Lambda → RDS Proxy → Aurora
  • Connection pooling and reuse
  • Automatic failover handling
  • IAM authentication support
  • Works with existing SQL clients
Strategy 2 — Simplest
Data API (HTTP)
Lambda → Data API → Aurora
  • No VPC required
  • No connection management
  • Higher latency per query
  • Aurora Serverless only
Strategy 3 — Not for Lambda
Direct Connection
App Server → Aurora
  • Long-running servers only (ECS, EC2)
  • Manage pool yourself (pg.Pool)
  • NOT suitable for serverless
  • PgBouncer sidecar for scale

RDS Proxy — IAM Authentication (TypeScript)
import { Signer } from '@aws-sdk/rds-signer';
import { Pool } from 'pg';

const signer = new Signer({
  hostname: process.env.RDS_PROXY_ENDPOINT,
  port: 5432,
  username: process.env.DB_USER,
  region: process.env.AWS_REGION
});

// Reuse pool across Lambda invocations
let pool: Pool | null = null;

export async function handler(event: any) {
  if (!pool) {
    const token = await signer.getAuthToken();
    pool = new Pool({
      host: process.env.RDS_PROXY_ENDPOINT,
      password: token,
      ssl: { rejectUnauthorized: true },
      max: 1,  // Single connection per Lambda
      idleTimeoutMillis: 120000
    });
  }
  return pool.query('SELECT ...', [event.id]);
}
Anti-Patterns
✗
Direct Lambda→Aurora connections — always use RDS Proxy or Data API; raw connections exhaust Aurora's connection limit
✗
No connection limits — set max: 1 in Lambda pg.Pool; use pooling in long-running servers
✗
No retry logic — handle transient errors from scaling/failover; use exponential backoff
✗
Missing IAM auth — use IAM over static passwords when possible; tokens rotate automatically
✗
Skipping Secrets Manager — never hardcode credentials; use automatic rotation with Secrets Manager
✗
Ignoring cold starts — keep min ACUs at 2+ for production; 0.5 ACU means cold-start latency on first query
