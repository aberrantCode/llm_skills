Spec Align

find spec → gap analysis → plan → implement → ship → verify

Given a feature spec name, filename, or topic, this workflow drives the codebase to full alignment — classifying every [XX-CAP-NN] capability as implemented, partial, or missing, writing a task-level plan, working each gap with strict TDD, and shipping a PR with a Spec Coverage table. Two human-gated checkpoints prevent autonomous implementation without explicit approval.
Input
  • Spec name, filename, or topic keyword
  • docs/features/ directory
  • docs/workflow/SDLC.md (abbreviation registry)
  • Existing codebase + test suite
→
Output
  • Aligned codebase — all CAP-IDs ✅
  • Passing test suite with named tests
  • Merged PR with Spec Coverage table
  • Spec Status → Implemented
  • Clean app startup verified
phases
Phase 0 Find the Spec
If given a filename — read directly. If given a topic — search docs/features/ for matching title, overview, or capabilities. Present candidates if multiple match; stop and ask if none found.
Phase 1 CAP-ID Check
Inspect Capabilities section for [XX-CAP-NN] prefixes. If any are missing → invoke homeradar-retro-fit-spec and commit the retro-fit alone. Only continue once all capabilities have prefixes.
Phase 2 Gate 1 — Spec Approval HUMAN REQUIRED
Check the spec's Status field. Must be Approved or In Progress to continue. Draft / In Review → surface to user, STOP. Implemented / Archived → confirm re-open. Cannot self-approve. Wait for explicit user confirmation.
Phase 3 Gap Analysis
For each [XX-CAP-NN] bullet and AC row: search tests + implementation files. Classify: ✅ Implemented (test + logic present), ⚠️ Partial (one missing), ❌ Missing (neither found). Output gap table. If all ✅ → stop and declare alignment — no plan needed.
Phase 4 Write Implementation Plan
For every ⚠️ / ❌ capability, write a B-style task contract. Save to docs/plans/YYYY-MM-DD-<spec-slug>-align-impl.md. Each task: CAP-ID, files to modify/create, exact test name, done-when condition, ordered steps. Ordering: schema → service/route → extension → web UI.
Phase 5 Gate 2 — Plan Approval HUMAN REQUIRED
Present gap table + plan. AskUserQuestion: Approved / Revise / Stop. No implementation code until user explicitly approves.
Phase 6 Implementation TDD
Create git worktree off latest dev. Set spec Status → In Progress. For each task: 🔴 RED — write named failing test, commit test: scaffold [XX-CAP-NN]. 🟢 GREEN — minimal impl to pass, commit with Refs: XX-CAP-NN. Check off AC row in spec. Never start next task until current test is green.
Phase 7 Pre-PR Gate
Invoke homeradar-pre-pr. All three gates must pass: full test suite for all touched stacks · lint + type-check (zero new errors) · every commit has Refs: line, no console.log, PR description has Spec Coverage table.
Phase 8 Ship to Dev PR
Invoke ship-to-dev. PR description must include Spec Coverage section: CAP-ID · Capability · AC Rows · Tests for each implemented gap. After merge: set spec Status → Implemented, commit directly to dev.
Phase 9 Start App
Run pwsh -NonInteractive -File scripts/Start-App.ps1. Verify the stack comes up cleanly. If any service fails to start, diagnose before declaring success.

hard rules
✕ Never self-approve Gate 1 or Gate 2 — both require explicit user confirmation before proceeding.
✕ Never write implementation code before Gate 2 passes.
✕ Never skip the RED (failing test) step for any task in the plan.
✓ Every implementation commit must include Refs: XX-CAP-NN — no exceptions.
✓ Retro-fit commits must be isolated — never bundled with implementation commits.
✕ If start-app fails, do not declare success — diagnose before closing out.
