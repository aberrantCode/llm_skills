release-to-main

Promotes dev → main as a versioned production release. Handles rebasing, automatic semantic versioning from conventional commits, merge commit, GitHub Release creation, and syncing dev back — with user confirmation at the version and merge steps.

Step 1 — Branch State Assessment
⛔
Nothing to release
AHEAD = 0
dev has no commits that main doesn't have. Stop immediately and tell the user.
↺
Rebase needed first
AHEAD > 0, BEHIND > 0
dev has new commits but is behind main. Must rebase dev onto main (Step 2) before merging.
✓
Ready to release
AHEAD > 0, BEHIND = 0
dev is ahead of main and already up-to-date. Skip Step 2 and go directly to Step 3.
Full Release Workflow
1
Fetch & Assess State Auto
git fetch origin. Count BEHIND (main commits dev lacks) and AHEAD (dev commits main lacks). If AHEAD=0 → STOP.
2
Rebase dev onto main Skip if BEHIND = 0
git checkout dev → stash → git rebase origin/main. Handle conflicts via AskUserQuestion (resolve manually or abort). After clean rebase: pop stash, git push --force-with-lease origin dev.
3
Determine Next Version Automatic + confirm
Get last tag via git describe --tags --abbrev=0. If no tags → first release from v0.0.0. Scan commits for bump signals. Present 3 candidates via AskUserQuestion with auto-detected recommendation.
4
Confirm the Release User confirmation required
Show summary: from branch, into branch, last release, new version, commit count. AskUserQuestion: "Yes — ship it" or "No — abort". If abort, stop cleanly with no git changes.
5
Merge dev into main Divergence check
git checkout main. Check LOCAL_AHEAD / LOCAL_BEHIND vs origin/main (see table below). git merge --no-ff origin/dev -m "release: $VERSION" → git push origin main.
6
Tag & Publish GitHub Release 3 guards active
Check tag doesn't already exist. git tag $VERSION && git push origin $VERSION. Detect repo with gh repo view --json nameWithOwner (never auto-detect). Generate notes from git log (only use range if PREV_TAG resolves). gh release create "$VERSION" --repo "$REPO".
7
Sync dev from main Auto
git checkout dev → git rebase origin/main (fast-forward, no conflicts) → git push --force-with-lease origin dev.
8
Show Final State Complete
git log --oneline -5 origin/main and git log --oneline -3 origin/dev. Report: PR complete, version tagged, dev synced.
Step 3 — Semantic Version Bump Rules
Pattern in commits Bump type Example: v1.2.3 becomes Note
BREAKING CHANGE in body, or feat!: / fix!: MAJOR v2.0.0 Highest priority — overrides everything
Subject starts with feat: or feat( MINOR v1.3.0 New functionality, backward-compatible
Anything else: fix:, chore:, refactor:, docs:, etc. PATCH v1.2.4 Lowest priority — default bump
No previous tags (IS_FIRST_RELEASE=true) from 0.0.0 v0.0.1 / v0.1.0 / v1.0.0 Scan all commits on dev (no git range)
Step 5 — Local main Divergence Handling
State LOCAL_AHEAD LOCAL_BEHIND Action
In sync 0 0 Proceed to merge
Behind only 0 >0 git pull origin main, then proceed
Ahead only (unpushed) >0 0 AskUserQuestion: push first or abort
Diverged >0 >0 STOP — do not merge, report to user
Critical Guards
No fake tag v0.0.0
Never use "v0.0.0" as a fallback string — it's not in git's object store. git log v0.0.0..HEAD exits 128. Use IS_FIRST_RELEASE flag instead.
Always pass --repo to gh
Never rely on gh auto-detection. Always set REPO=$(gh repo view --json nameWithOwner -q .nameWithOwner) and pass --repo "$REPO".
Validate PREV_TAG before range
Before running git log PREV_TAG..VERSION, verify PREV_TAG resolves: git rev-parse "$PREV_TAG" >/dev/null 2>&1. Fall back to all commits if not.
Tag existence check
Before pushing a tag, run git rev-parse "$VERSION" >/dev/null 2>&1. If tag exists → AskUserQuestion: different version or abort.
Never force-push main
If push to main is rejected: git pull --rebase origin main then retry. Never --force on main.
Release notes fallback
If notes are empty after git log, set fallback: "(release notes unavailable — check git log manually)". Never suppress with 2>/dev/null.
Integration
Promotes: dev → main
Requires: gh CLI authenticated
Feeds: ship-to-dev (keeps branches alive)
Creates: GitHub Release (required for /releases/latest API)
