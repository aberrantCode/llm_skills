Partition Key Design
Good — User Data
partitionKey: /userId
High cardinality, even distribution, frequently used in queries. Each user's data lands in its own logical partition.
Good — Multi-tenant
partitionKey: /tenantId
Tenant-scoped queries are single-partition. Use hierarchical keys (/tenantId, /userId) for sub-partitioning.
Bad — Low Cardinality
partitionKey: /status
Only 3-5 unique values creates hot partitions. All "pending" orders go to one partition, exhausting its 10K RU/s limit.

CRUD Patterns + Change Feed
Point Read (most efficient)
// 1 RU — requires id + partition key
const { resource } = await
  usersContainer
    .item(userId, userId)
    .read<User>();
return resource || null;

// Patch operations (partial update)
await usersContainer
  .item(id, userId)
  .patch([
    { op: 'set', path: '/name',
      value: 'New Name' },
    { op: 'incr', path: '/loginCount',
      value: 1 }
  ]);

  const { result, statusCode }
    = await iterator.fetchNext();

    // No new changes
    await sleep(1000); continue;
  }

  for (const item of result) {
    // Process change event
  }
}

Time-to-Live (TTL)
// Container-level default TTL
await db.containers
  .createIfNotExists({
    id: 'sessions',
    partitionKey: { paths: ['/userId'] },
    defaultTtl: 3600  // 1 hour
  });

// Per-item TTL override
const session = {
  id: 'sess-123',
  userId: 'user-456',
  ttl: 1800  // 30 minutes
};
Anti-Patterns
✗
Bad partition key — low cardinality causes hot partitions; status/type/boolean fields are never good partition keys
✗
Cross-partition queries everywhere — expensive; design for single-partition queries and use hierarchical keys
✗
Over-indexing — each indexed path adds write RU cost; index only fields used in WHERE clauses
✗
Strong consistency everywhere — use Session (default) unless financial or inventory accuracy requires Strong
✗
No retry for 429s — implement exponential backoff; 429 (throttling) is normal at burst; handle gracefully
✗
Missing TTL for temporary data — always set TTL on sessions, tokens, and cache documents to avoid unbounded growth
