Skill Reference

Database Schema Awareness

Read the schema before writing any database code. Verify column names and types. Use generated types. Prevent column-name errors before they reach runtime.

Read First Type Generation TDD-Schema
MANDATORY RULE
Before writing ANY code that touches the database:
1. READ the schema file
2. VERIFY columns/types you are about to use exist
3. REFERENCE schema in your response when writing queries
4. TYPE-CHECK using generated types (Drizzle / Prisma / etc.)
If schema file doesn't exist → CREATE IT before proceeding.

Common Schema Mistakes
MistakeExamplePrevention
Wrong column name user.userName Read schema; use generated types
Wrong type totalCents as string Type generation catches this
Missing nullable check user.name! Schema shows nullable columns
Wrong FK reference order.userId vs order.user_id Check schema column names
Wrong enum value 'complete' vs 'completed' Document enums in schema reference

Anti-Patterns
✗
Guessing column names — always read the schema file first; never assume a column name
✗
Using raw SQL strings — use ORM with type generation; raw strings have no compile-time verification
✗
Ignoring type errors — schema drift shows up as TypeScript/mypy errors; fix them, don't suppress
✗
Not regenerating types after migration — stale types hide real schema; always regenerate after every migration
✗
Assuming nullable — check the schema explicitly; a column that looks optional may be NOT NULL with a default
