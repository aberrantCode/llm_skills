Requesting
Code Review

superpowers:requesting-code-review

Dispatch a code-reviewer subagent at the right moment to catch issues before they cascade. Early review is always cheaper than debugging later — review after each task, not just before merge.

Severity Triage — How to Act on Feedback
Severity Action Required Can Proceed? Example
Critical Fix immediately No Security hole, data loss, broken contract, test-breaking regression
Important Fix before proceeding No Missing error handling, wrong abstraction, silent failure path
Minor Note for later Yes Magic number, naming preference, minor style inconsistency
None / Approved Proceed Yes Reviewer found no blocking issues — continue to next task
When to Request Review
Mandatory
  • After each task in subagent-driven development
  • After completing a major feature
  • Before merge to main
Optional but Valuable
  • When stuck — fresh perspective unblocks thinking
  • Before refactoring — establish baseline
  • After fixing a complex bug
Integration with Workflows
Subagent-Driven Dev
After every task
Review after each individual task. Catch issues before they compound into the next task's foundation.
Executing Plans
After each batch (3 tasks)
Review after every batch of 3 tasks. Apply feedback, then continue with the next batch.
Ad-Hoc Development
Before merge / when stuck
Review before merging to main, or whenever you're stuck and want a fresh perspective on the code.
Key Rules
🚫
Never skip "simple" tasks
Simple tasks have simple bugs. The mandatory cadence exists precisely because "simple" is often wrong.
🚫
Never ignore Critical
Critical issues block everything. No proceeding to the next task until they are fully resolved.
🚫
Never argue with valid feedback
If the feedback is technically correct, fix it. Defensiveness compounds issues rather than resolving them.
✓
Push back if reviewer is wrong
Show the code or tests that prove it works. Request clarification. Technical reasoning always wins over blind agreement.
✓
Fix Important before proceeding
Important issues aren't optional. Fix them before moving to the next task or feature — don't carry debt forward.
✓
Note Minor, revisit later
Minor issues don't block forward progress, but track them. Clean up during a refactor pass or before final merge.
