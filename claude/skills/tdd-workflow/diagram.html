Test-Driven Development

Write tests first. Make them pass. Refactor. Always. Tests are not optional — they are the safety net that enables confident refactoring and production reliability.

Minimum 80% coverage required

02 — Full TDD Workflow (7 Steps)
1
Write User Journey
Define what the user needs: "As a user, I want to search for markets, so I can find relevant results without exact keywords."
2
Generate Test Cases
Write comprehensive tests covering happy path, edge cases, errors, boundary conditions, and fallback behaviors. One assertion per test.
3
Run Tests — They Must FAIL
Confirm the tests actually fail before writing any implementation. Failing tests prove the tests are meaningful.
npm test # → FAIL (expected)
4
Implement Code
Write the minimal implementation to make the tests pass. Don't over-engineer at this stage.
5
Run Tests — They Must PASS
All tests should now be green. If not, fix the implementation (not the tests — unless the tests were wrong).
npm test # → PASS ✓
6
Refactor
Improve code quality: remove duplication, improve naming, optimize performance, enhance readability. Tests stay green throughout.
7
Verify Coverage
Must achieve 80%+ on branches, functions, lines, and statements. Identify gaps and add tests as needed.
npm run test:coverage

05 — Testing Patterns
✅ Correct Patterns
Test user-visible behavior, not internal state
One assertion per test — focused & clear
Each test sets up its own data (no shared state)
Use semantic selectors: role, text, testId
Mock external dependencies (DB, Redis, APIs)
Arrange-Act-Assert structure in every test
❌ Anti-Patterns
Testing component.state.count (internal)
Tests depending on execution order
CSS selectors like .css-class-xyz
Skipping or disabling failing tests
waitForTimeout() in E2E tests
Fixing tests to match broken implementation
